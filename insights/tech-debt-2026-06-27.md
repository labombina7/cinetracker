## Hallazgos

### 🔒 Seguridad

#### OAuth code y clientId expuestos en logs
- **Fichero:** `src/services/trakt/auth.ts:29-33`
- **Severidad:** 🔴 Alta
- **Descripción:** `console.log` imprime el código OAuth de autorización y el `clientId` en texto plano. El código de autorización es un secreto de un solo uso; cualquier extensión de browser o log server podría capturarlo.
- **Fix propuesto:** Eliminar todos los `console.log` con datos de auth en producción. Usar un logger condicional (`if (import.meta.env.DEV)`) o eliminarlos completamente.

#### Token response logueado en texto plano
- **Fichero:** `src/services/trakt/auth.ts:67`
- **Severidad:** 🔴 Alta
- **Descripción:** `console.log('Token response:', data)` imprime el objeto completo incluyendo `access_token` y `refresh_token` en la consola del navegador.
- **Fix propuesto:** Eliminar esta línea. Si se necesita debugging, loguear solo `data.token_type` o un boolean indicando éxito.

#### State CSRF generado con Math.random()
- **Fichero:** `src/services/trakt/auth.ts:12`
- **Severidad:** 🟡 Media
- **Descripción:** El parámetro `state` de OAuth se genera con `Math.random().toString(36)`, que no es criptográficamente seguro. Un atacante con acceso al generador de números pseudo-aleatorios podría predecir el valor.
- **Fix propuesto:** Usar `crypto.getRandomValues(new Uint8Array(16))` y convertirlo a hex para un state verdaderamente aleatorio.

#### TMDB API key en localStorage sin cifrado
- **Fichero:** `src/components/ApiKeySetup.tsx:28`
- **Severidad:** 🟡 Media
- **Descripción:** La API key de TMDB se guarda en `localStorage` como texto plano. Cualquier script de la página (incluyendo extensiones) puede leerla.
- **Fix propuesto:** Para una app personal esto es aceptable, pero documentarlo. Considerar usar `sessionStorage` para que no persista entre sesiones si la seguridad es prioritaria.

#### Trakt client_secret en localStorage
- **Fichero:** `src/services/trakt/config.ts:12`
- **Severidad:** 🟡 Media
- **Descripción:** El `client_secret` de Trakt se recupera directamente de `localStorage`. Para apps de cliente puro esto es un anti-patrón de OAuth; el secret nunca debería estar en el cliente.
- **Fix propuesto:** Usar el flujo PKCE de OAuth 2.0 en lugar de client_secret para apps SPA públicas. El endpoint de Trakt lo soporta.

### ⚠️ Gestión de errores

#### Colisión de IDs en favoritos
- **Fichero:** `src/hooks/useFavorites.ts:22-68`
- **Severidad:** 🔴 Alta
- **Descripción:** `favorites` es un array de `number[]` (solo IDs). Una película con ID 123 y una serie con ID 123 son indistinguibles. TMDB reutiliza IDs entre movies y TV shows.
- **Fix propuesto:** Cambiar la estructura a `{ id: number; type: 'movie' | 'tv' }[]` y adaptar `isFavorite` para recibir ambos parámetros.

#### fetchMediaDetails retorna null silenciosamente
- **Fichero:** `src/services/tmdb/details.ts:114-117`
- **Severidad:** 🟡 Media
- **Descripción:** El catch retorna `null` sin propagar el error. Los componentes que llaman a esta función deben manejar `null` pero no reciben información sobre qué falló.
- **Fix propuesto:** Lanzar el error o retornar un `Result<Media, Error>`. Como mínimo, re-throw para que el componente pueda mostrar un error específico.

#### getAccessToken retorna null en vez de lanzar
- **Fichero:** `src/services/trakt/auth.ts:63,79`
- **Severidad:** 🟡 Media
- **Descripción:** Tanto en errores de red como en respuesta no-OK, la función retorna `null`. El llamador (`AuthCallback.tsx`) no puede distinguir entre "token nulo" y "error de red", ni mostrar un mensaje específico.
- **Fix propuesto:** Definir tipos de error concretos o lanzar excepciones tipadas para que el callback las maneje de forma diferenciada.

#### JSON.parse de favorites sin validación
- **Fichero:** `src/hooks/useFavorites.ts:13`
- **Severidad:** 🟡 Media
- **Descripción:** `JSON.parse(storedFavorites)` sin validar que el resultado sea un `number[]`. Si localStorage está corrupto o fue manipulado, el app puede crashear o comportarse de forma inesperada.
- **Fix propuesto:** Validar con `Array.isArray(parsed) && parsed.every(n => typeof n === 'number')` antes de usar el valor parseado.

#### HomeErrorState muestra mensajes técnicos al usuario
- **Fichero:** `src/components/home/HomeErrorState.tsx:10-13`
- **Severidad:** 🟡 Media
- **Descripción:** El prop `error` puede ser cualquier string técnico de la API (ej. "Error fetching trending: 401") que se muestra directamente al usuario.
- **Fix propuesto:** Mapear errores técnicos a mensajes de usuario amigables en `fetchMediaError.ts` antes de que lleguen al componente.

### 🏷️ Tipos TypeScript

#### fetchMediaPlatforms con parámetros any
- **Fichero:** `src/hooks/mediaFetch/fetchMediaPlatforms.ts:7-8`
- **Severidad:** 🔴 Alta
- **Descripción:** `fetchMediaByStrategy: any` y `currentParams: any` eliminan toda la seguridad de tipos en la función crítica de fetching de plataformas.
- **Fix propuesto:** Definir interfaces específicas para estos parámetros. `currentParams` ya tiene una forma implícita que puede extraerse a un tipo `FetchParams`.

#### useFetchMedia sin tipado en sus parámetros de desestructuración
- **Fichero:** `src/hooks/mediaFetch/useFetchMedia.ts:9-24`
- **Severidad:** 🟡 Media
- **Descripción:** La función del hook recibe sus dependencias por desestructuración sin tipo explícito, resultando en tipos inferidos como `any` para varias de ellas.
- **Fix propuesto:** Definir una interfaz `UseFetchMediaDeps` con todos los parámetros tipados y usarla en la firma del hook.

#### TraktUserProfile con userInfo: any

- **Descripción:** El prop `userInfo` es `any`. Accesos como `userInfo.movies?.watched` podrían fallar silenciosamente en runtime si la respuesta de Trakt cambia.

#### Servicios Trakt con retorno any

#### Media.platform duplicado como deprecated
- **Fichero:** `src/types/media.ts:13`
- **Severidad:** 🟢 Baja
- **Descripción:** `platform?: Platform` se mantiene "para compatibilidad con código existente" pero crea ambigüedad con `platforms?: Platform[]`.
- **Fix propuesto:** Hacer una búsqueda de usos de `media.platform` (singular) y migrarlos a `platforms`. Eliminar el campo deprecated.

#### item.genres.map con any en details
- **Fichero:** `src/services/tmdb/details.ts:30`
- **Severidad:** 🟢 Baja
- **Descripción:** `item.genres.map((g: any) => ...)` y múltiples `forEach(provider => ...)` sin tipos en el mismo fichero.
- **Fix propuesto:** Crear interfaces TMDB crudas (`TMDBMovie`, `TMDBProvider`) para los objetos de la API antes de transformarlos.

### 🚀 Rendimiento

#### fetchGenres llamado por cada MediaCard
- **Fichero:** `src/components/MediaCard.tsx:36-63`
- **Severidad:** 🔴 Alta
- **Descripción:** Cada `MediaCard` llama a `fetchGenres(mediaType)` en su `useEffect` al montar. Con 20 cards en pantalla, esto genera hasta 40 peticiones a TMDB (20 movie + 20 tv) en el mismo render cycle, aunque el endpoint devuelva los mismos datos.
- **Fix propuesto:** Elevar la carga de géneros a un hook compartido (`useGenres` ya existe en `src/hooks/useGenres.ts`) y pasar los géneros como prop a `MediaCard`, o usar el caché de `apiCache.ts` que ya existe pero no es llamado desde `fetchGenres`.

#### Imágenes sin lazy loading
- **Fichero:** `src/components/MediaCard.tsx:84`, `src/components/media/MediaCastCarousel.tsx:27`
- **Severidad:** 🟡 Media
- **Descripción:** Las imágenes de póster y de actores no tienen el atributo `loading="lazy"`. En una lista larga, el browser carga todas las imágenes inmediatamente.
- **Fix propuesto:** Añadir `loading="lazy"` a todos los `<img>` de listas. En el carrusel de cast añadir también `decoding="async"`.

#### URL de imagen de TMDB hardcodeada (no usa cachedFetch)
- **Fichero:** `src/components/media/MediaCastCarousel.tsx:28`
- **Severidad:** 🟡 Media
- **Descripción:** La URL de imagen de los actores está hardcodeada como `https://image.tmdb.org/t/p/w185${actor.profile_path}` en lugar de usar la constante de configuración.
- **Fix propuesto:** Usar `TMDB_CONFIG.IMAGE_BASE_URL` y `TMDB_CONFIG.POSTER_SIZES.SMALL` para consistencia y facilitar cambios futuros.

#### Cache in-memory se pierde en cada navegación
- **Fichero:** `src/services/tmdb/apiCache.ts`
- **Severidad:** 🟡 Media
- **Descripción:** El cache de API es un `Map` en memoria que se reinicia con cada carga de página. El usuario que navega de Home → Detail → Home hace todas las peticiones de nuevo.
- **Fix propuesto:** Para páginas de detalle, persistir el resultado en `sessionStorage` con TTL. El hook `useMediaSession` ya hace esto para la lista principal; extender el patrón.

### 📡 Resiliencia

#### Token de Trakt sin refresco automático
- **Fichero:** `src/services/trakt/auth.ts:71-73`
- **Severidad:** 🔴 Alta
- **Descripción:** El `refresh_token` se guarda en localStorage pero no hay ningún mecanismo de refresh automático. Cuando el `access_token` expire (típicamente en 3 meses para Trakt), el usuario verá errores 401 sin explicación clara.
- **Fix propuesto:** Implementar una función `refreshAccessToken()` que use el `refresh_token` guardado, y llamarla en los interceptores de las peticiones que fallen con 401.

#### sessionStorage puede alcanzar el límite de 5MB
- **Fichero:** `src/hooks/useMediaSession.ts:13-14`
- **Severidad:** 🟢 Baja
- **Descripción:** Arrays grandes de resultados de media se guardan en sessionStorage sin verificar el tamaño disponible. En catálogos grandes, esto podría fallar silenciosamente (el `catch` en `saveResultsToSession` lo silencia).
- **Fix propuesto:** Limitar el número de items guardados (ej. máximo 100) o verificar el tamaño antes de escribir.

#### Sin fallback cuando TMDB no responde en detalle
- **Fichero:** `src/services/tmdb/details.ts:114`
- **Severidad:** 🟡 Media
- **Descripción:** Si TMDB devuelve error al cargar los detalles de un título, `fetchMediaDetails` retorna `null` y la página de detalle queda en estado de carga indefinido o muestra contenido vacío.
- **Fix propuesto:** Los componentes que llaman a `fetchMediaDetails` deben manejar explícitamente el caso `null` con un error state con opción de retry.
